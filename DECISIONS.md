# Decisions

Log of choices not fully specified by INSTRUCTIONS.md / SPEC.md, per the "note the decision" rule.

- **2026-07-23 — Product name/folder:** `alonapp` after the chosen domain alonapp.com; product name "Alon".
- **2026-07-23 — Next.js 16.2:** `create-next-app@latest` produced Next 16 (spec says 14+). Kept — newest App Router, TS strict on by default.
- **2026-07-23 — RLS tests in Vitest, not pgTAP:** TypeScript suite (`tests/rls.test.ts`) using `@supabase/supabase-js`; service-role client seeds fixtures, per-role user clients assert isolation. Same `npm test` entry point locally and in CI, no extra pg tooling.
- **2026-07-23 — Supabase CLI as npm devDependency:** no global install needed; `npx supabase …` works identically locally and in CI.
- **2026-07-23 — Client JWT claims via custom access token auth hook:** Postgres function `public.custom_access_token(event jsonb)` reads `public.user_roles` and injects `user_role` + `client_id` claims; registered in `supabase/config.toml`. Freelancers get `user_role: 'freelancer'` on workspace creation; client users get `user_role: 'client'` + their `client_id`.
- **2026-07-23 — Client file uploads deferred:** INSTRUCTIONS.md's RLS section says client sessions may INSERT on `messages` only (and the merge-required tests assert exactly that), so storage/table policies give clients read-only file access. `files.uploaded_by = 'client'` stays in the schema for when uploads land in v2.
- **2026-07-23 — JWT claim named `user_role`, not `role`:** the reserved `role` claim maps to the Postgres role (`authenticated`) and can't safely carry `'client'`; policies check `auth.jwt() ->> 'user_role'` instead.
- **2026-07-23 — Production deployed early (before milestone 3):** hosted Supabase project `fvsupccynnhqeywpuukg` in `ap-southeast-1` (Singapore, closest region to PH); Vercel project `alonapp` with GitHub push-to-deploy from `judegamboa/alonapp` main; live at https://www.alonapp.com (apex alonapp.com redirects to www; alonapp.vercel.app also serves). All 5 migrations pushed (7 as of milestone 9 — `branding_bucket` and `billing_columns` came later); auth config (custom access token hook) synced via `supabase config push`. DB password lives with the owner only (resettable in the dashboard), service-role key only in Vercel production env.
- **2026-07-23 — Email debounce is per-thread, direction-agnostic:** `notifyNewMessage` skips if the thread's `last_emailed_at` is within 15 min, regardless of who sent the last message. Recipient is chosen by author: a freelancer message notifies the client's email; a client message notifies the workspace owner's email. `lib/email.ts` no-ops (logs) when `RESEND_API_KEY` is unset so local flows never break.
- **2026-07-23 — freelancer-auth email template exists but Supabase sends the live ones:** the 6th React Email template is built, but Supabase Auth still delivers freelancer confirmation/reset mail (Mailpit locally). Wiring our template requires a Supabase "send email" auth hook — deferred; the other 5 templates fire from our own server actions.
- **2026-07-23 — Local dev via Supabase CLI (Docker):** ~~no hosted project yet; hosted project + Vercel wiring deferred to deployment.~~ **Superseded** by the production-deploy entry above — the hosted project and Vercel wiring landed before milestone 3. The Docker-based local stack remains the dev workflow.
- **2026-07-23 — A 7th email template (`payment-request`):** SPEC.md locks 6 templates and none covers payment requests, but a request the client never learns about is useless — the portal is pull-only. Added `emails/payment-request.tsx` + `notifyPaymentRequest`, fired once on creation (no debounce; requests are deliberate, low-frequency acts, unlike messages). The CTA deep-links into the portal rather than the pasted payment URL, matching `files-shared`'s "link into the portal, never the payload" rule.
- **2026-07-23 — "Mark as paid" is freelancer-only:** SPEC.md says "manual mark as paid toggle" without naming who toggles. The milestone-2 RLS grants clients `SELECT` on `payment_requests` and nothing else, and INSTRUCTIONS.md scopes client writes to messages — so the freelancer marks payment received. `tests/rls.test.ts` now pins this (client update/delete both affect 0 rows).
- **2026-07-23 — Currency as short dropdown + free code:** USD/PHP/AUD cover the corridor in SPEC.md (PH freelancers, US/AU clients); a 3-letter `currency_other` field overrides the dropdown for everyone else. `formatAmount` (`lib/currency.ts`) wraps `Intl.NumberFormat` and falls back to `CODE 0.00` for codes Intl rejects, so an odd currency degrades to plain text instead of throwing.
- **2026-07-23 — Mobile overflow came from unbroken strings, not layout:** measuring the portal constrained to 390px showed 11 overflowing elements, all traced to two sources — pasted URLs in message bodies (266px box, 614px content) and long filenames. The flex/grid layout itself was already fine. Fixed with `break-words` on message bodies and payment descriptions, `break-all` on filenames and client emails, plus `min-w-0` on the flex parents that were refusing to shrink. Worth noting the diagnostic: comparing `getBoundingClientRect()` widths finds nothing, because the *box* stays put while the *content* spills — `scrollWidth > clientWidth` is the check that catches it.
- **2026-07-23 — Portal `loading.tsx` is deliberately unbranded:** the workspace's `brand_color` isn't known until the portal row loads, and flashing Alon's teal before the freelancer's color would undercut the white-label promise. The skeleton stays neutral and the brand bar fills in on paint.
- **2026-07-23 — Billing columns revoked from `authenticated` (security fix):** `workspaces` had an "owner full access" RLS policy plus a blanket table grant, so a freelancer could run `supabase.from('workspaces').update({ plan: 'pro' })` with the public anon key and promote themselves — every tier limit was decorative. RLS can't scope to columns, so `20260723000700_billing_columns.sql` revokes INSERT/UPDATE on the table from `authenticated` and grants back only `name`, `logo_url`, `brand_color` (plus `owner_id` on insert). Note a column-level `revoke` alone would have done nothing: Postgres treats table-level and column-level privileges separately, so the table grant had to be dropped first. `plan` and the `paddle_*` columns are now service-role-only, i.e. the webhook. `tests/rls.test.ts` pins both the denial and that branding edits still work.
- **2026-07-23 — Paddle confirmed over Lemon Squeezy:** SPEC.md left it open until week 5; the schema already carries `paddle_customer_id` / `paddle_subscription_id`, so switching would have cost a rename migration for no gain.
- **2026-07-23 — Webhook signature verified with node `crypto`, not `@paddle/paddle-node-sdk`:** with no Paddle account yet, an SDK we can't exercise adds a dependency and hides the one piece that must be right. `lib/paddle.ts` computes HMAC-SHA256 over `${ts}:${rawBody}` and compares with `timingSafeEqual` (length-checked first, since it throws on mismatch), rejecting signatures older than 5 minutes. `tests/paddle-signature.test.ts` covers tamper/wrong-secret/replay/malformed with zero new deps — the suite is node-environment, so no jsdom was needed. The route reads `request.text()` before parsing; calling `request.json()` first would make the raw bytes unrecoverable and the signature unverifiable.
- **2026-07-23 — Dev-only plan switcher, double-gated:** with no Paddle keys nothing could move a plan locally, so tier limits were untestable. Settings renders a plan selector only when `NODE_ENV !== 'production'` **and** `paddleConfigured()` is false, and `setPlanForDev` re-checks both itself — hiding a form doesn't stop anyone posting to the action. It writes via service-role because `authenticated` no longer has the grant. Both guards are load-bearing: without them this is a free-upgrade endpoint.
- **2026-07-23 — Downgrades are soft:** a Pro workspace with 12 portals dropping to Free keeps all 12 working and is simply blocked from creating more (`atLimit` uses `>=`, so the dashboard reads "12 of 1 used"). Auto-archiving the excess would cut a freelancer's client off from their portal with no warning.
- **2026-07-23 — Pro's "custom portal subdomain" is advertised but unbuilt:** it appears in the locked pricing table (`docs/SPEC.md`), the landing-page pricing card (`app/page.tsx`), and the plan-limit copy (`lib/workspace.ts`), but nothing implements it — portals are path-based (`/p/[portal_slug]`) and there is no wildcard-domain handling in `proxy.ts` or Vercel. Deferred to v2: it needs wildcard DNS, per-workspace domain records, and certificate provisioning, none of which the MVP scope funded. Flagged here because it is the one place the marketing copy promises something the app cannot do — the copy needs to change or the feature needs building before Pro is sold. Free's app-branded watermark, the other pricing-table claim, *is* implemented (`app/p/[portal_slug]/page.tsx`, gated on `plan === 'free'`).
- **2026-07-23 — "Custom portal subdomain" pulled from all user-facing copy, then re-added as "coming soon":** acting on the flag in the entry above, the claim was briefly removed entirely from the landing pricing card (`app/page.tsx`) and `PLAN_BLURBS.pro` (`lib/workspace.ts`). Reinstated same day at the owner's call, now suffixed "(coming soon)" in both places — it stays roadmap-visible on Pro without claiming it ships today. `docs/SPEC.md` already carries the *(not built — see DECISIONS.md)* annotation, so all three copies now agree. **Drop the "(coming soon)" suffix only once wildcard DNS and per-workspace domains actually exist.**
- **2026-07-23 — Paid-tier CTAs say "Start free, upgrade later":** Paddle checkout is unwired (webhook-in only), so a "Get started" button on Starter/Pro sent people to signup and then a dead end. Until Paddle.js lands, paid cards route to signup with a label that describes what actually happens, plus a line under the grid: "Everyone starts on Free. Paid plans open up as Alon leaves beta." Revert to a checkout link when the flow exists.
- **2026-07-23 — Client portals are `noindex`, belt and braces:** `app/robots.ts` disallows `/p/`, `/app/`, `/auth/` and `/api/`, and `app/p/[portal_slug]/page.tsx` also exports `robots: { index: false, follow: false }`. robots.txt is a request, not a control, and a portal slug is guessable enough without search engines helping — the page-level directive covers crawlers that reach the URL some other way (a client pasting the link somewhere public).
- **2026-07-23 — OG image generated in code, font committed:** `app/opengraph-image.tsx` builds the 1200×630 card with `next/og` rather than shipping a static PNG, so the headline and tideline stay in step with the site. Satori cannot read CSS variables or woff2, so the palette hexes are restated in that file (keep them in step with `:root` in `globals.css`) and a static Bricolage Grotesque 700 TTF is committed at `assets/bricolage-grotesque-700.ttf` — fetching it from Google Fonts at build time would make the build fail on a network blip.
- **2026-07-23 — Taglish wink pulled from marketing copy:** the landing page's "saan na tayo?" line (the alon-design skill's one-permitted-wink example) read wrong once the client-not-signup story made clear most clients are overseas and never see Tagalog elsewhere on the page. Replaced with an English line carrying the same beat ("One link. Your client checks it instead of checking on you."); `alon-design/SKILL.md`'s copy-voice rule updated to keep marketing English throughout rather than allow one wink per page.
- **2026-07-23 — Milestone 8 needed no migration:** `payment_requests` (table, index, RLS policies, grants) shipped back in milestone 2 alongside the rest of the schema; milestone 8 was app-layer only.
